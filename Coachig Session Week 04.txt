/**
 * Requirement 3: We then process the Edit Category Form Submission with Server-side validation check
 */
export const processEditCategoryForm = async (req, res, next) => {
    const categoryId = parseInt(req.params.id, 10);
    try {
        const errors = validationResult(req);
        if (!errors.isEmpty()) {
            errors.array().forEach(error => req.flash('error', error.msg));
            return res.redirect(`/edit-category/${categoryId}`);
        }

        const {name} = req.body;
        await categoryModel.updateCategory(categoryId, name);

        req.flash('success', 'Category updated successfully!');
        res.redirect(`category/${categoryId}`); // This redirects back to single category detail view
    } catch (error) {
        console.error('Error in processEditCategoryForm:', error);
        next(error);
    };
}

Let's move on to our third mandatory topic: Sessions and Flash Messages. In your very first code snippet, I noticed you used 'req.flash('error', error.msg)' and 'req.flash('success', 'Category updated successfully!')'. Could you explain why we store flash messages inside a session, and what would happen if we tried to pass them to the next page using a standard local variable instead of a session-backed flash utility?

[STUDENT]: Flash messages are stored inside a session because they are meant to survive one HTTP request cycle — especially during redirects.

[COACH]: Spot on, Garikai! You nailed the core architectural reason. Because HTTP is stateless, the moment your controller sends a redirect response ('res.redirect'), the current request/response cycle ends completely. If you used a standard local variable, that data would instantly vanish into thin air before the new page even loads. By storing the message in the session, it stays safely on the server across that redirect boundary and is deleted automatically the moment the next page renders it.

Looking at your 'projectValidation' array, you have this rule for the organization ID:

Why is it so vital to validate that 'organizationId' is strictly an integer ('.isInt()') before passing it down to your database query function? What kind of trouble could occur if we skipped that specific type check?

[STUDENT]: When we use .isInt() for validating the organizationId strictly as an integer, is important because this protects the databse from invalid or malicious input.

[COACH]: Exactly, Garikai! It is your ultimate safeguard for database integrity. When you enforce '.isInt()', you are blocking two major issues:
1. Database Crash/Type Errors: If a user somehow submits a string like "apple" instead of an integer like 2, passing that straight into an SQL query expecting an integer would cause the database driver to throw a fatal error and potentially crash your server application.
2. SQL Injection Security Risks: If a malicious user attempts to insert SQL syntax or malicious code into that form field, treating it strictly as an integer prevents the database engine from executing it as a command.
